# 银行版 OpenClaw 的 PRD 雏形

## 一、产品定义

### 产品名称

暂定名：`Banking Assistant OS`

### 产品定位

一个面向银行员工的、绑定工作身份的 AI Assistant 平台。  
它不是个人助理，也不是通用聊天机器人，而是：

`[OpenClaw 借鉴点]`  
`[银行化改造]`

- 绑定员工身份与岗位权限
- 围绕任务与流程工作
- 调用受控技能与工具
- 具备持续上下文和主动能力
- 可向 Agent-native 和数字岗位演进

这里的“任务”不是狭义 IT 系统中的待办节点，而是：

**银行员工围绕某个业务对象执行的一组真实工作动作的 AI 管理单元。**

因此，这个产品的核心不是给每个员工一个聊天入口，而是给每个员工一个：

- 工作身份绑定的 Assistant
- 工作语义任务层
- 受控技能与工具层
- 可编排、可审计、可沉淀的工作代理系统

### 产品愿景

短期内，让银行员工在高频工作中“更强、更快、更稳”；  
中期，把高频岗位劳动沉淀为可治理的自动化能力；  
长期，将部分成熟能力制度化为数字岗位，承接组织级产能。

### OpenClaw 借鉴点标记说明

为了便于后续阅读，这份 PRD 中会使用如下标记：

- `[OpenClaw 借鉴点]`：表示该能力或设计思路直接借鉴了 OpenClaw 的核心产品哲学
- `[银行化改造]`：表示该能力虽然受 OpenClaw 启发，但已经按银行场景做了治理、权限或组织化改造

本 PRD 里显式借鉴 OpenClaw 的核心点，主要包括：

1. Assistant 不是页面，而是常驻工作体
2. 持续上下文与分层记忆
3. 技能 / 工具调用是一等能力
4. 主动能力，而不只是被动问答
5. 多触点存在，而不是单一应用入口
6. AI 更像工作代理，而不只是聊天工具

## 二、目标用户

## 1. 第一阶段核心目标用户

建议优先聚焦四类用户：

### A. 预审 / 初筛 / 初核岗位

典型场景：

- KYC 预审
- 贷前资料预审
- 交易告警初筛
- 标准化前处理

原因：

- 任务重复度高
- 输出相对标准
- Assistant 价值明确
- Agent 化潜力高

### B. 运营处理岗位

典型场景：

- 标准工单处理
- 材料核验
- 运营后处理
- 例行核对任务

原因：

- 岗位规模大
- 替代密度高
- 适合承接 headcount 压降目标

### C. 授信审批辅助岗位

典型场景：

- 材料归纳
- 风险点摘要
- 审查意见草稿

原因：

- 能体现 Assistant 的认知价值
- 能帮助测绘复杂岗位劳动结构
- 能反哺前处理和预审能力沉淀

### D. 银行客服中心标准服务岗位

典型场景：

- 标准业务问答
- 业务流程引导
- 资料补充提醒
- 标准工单创建与流转
- 会话摘要与后续动作建议

原因：

- 人员规模大
- 任务标准化程度高
- 知识与话术可沉淀
- Assistant 使用频率极高
- Agent 化潜力强
- 适合承接 headcount 压降目标

## 2. 第二阶段扩展用户

- 风控分析岗
- 合规审查岗
- 客户经理
- 复杂投诉处理坐席
- 客服质检与运营支持人员

## 3. 长期用户范围

全行正式员工、部分集中作业岗位、部分数字岗位运营人员。

## 三、核心场景

## 1. 场景：复杂任务规划与统一交付

默认情况下，用户发起的每个 `Session` 都应被视为复杂任务，而不是先假设它只是单一步骤请求。

例如：

- 先筛客户，再排序，再生成会前摘要
- 先检查缺件，再识别高风险件，再生成补件单
- 帮我看这批件哪些要优先跟进，并起草补件话术

在这个主场景中，系统不应把“上下文整理”“岗位动作协助”“主动提醒”“预处理承接”“长尾托底”理解为彼此独立的场景，而应将它们视为同一复杂任务中的不同处理阶段。

### 系统内部过程

#### A. 理解与路由

系统先基于当前 `Session`、页面上下文、业务对象、岗位身份和历史轨迹，判断：

- 当前工作意图属于哪个任务模板或任务组合
- 是否需要新建 `Task`，还是挂接已有 `Task`
- 是否存在需要关联的历史 `Session`

#### B. 上下文整理

一旦进入任务处理，Assistant 自动整理：

- 业务对象信息
- 相关材料
- 历史处理记录
- 政策依据
- 风险点摘要
- 下一步待办

#### C. 任务规划

在执行前，系统先生成任务计划，明确：

- 目标结果是什么
- 需要拆成哪些子任务
- 哪些步骤可并行，哪些步骤有先后依赖
- 哪些步骤可以自动执行，哪些步骤必须人工确认
- 最终以什么结构统一交付

#### D. 岗位动作协助

围绕当前任务计划和子任务节点，系统提供结构化岗位动作支持，例如：

- 材料完整性检查
- 风险点归纳
- 审查意见草稿生成
- 补件清单生成
- 标准回复草稿生成

#### E. 主动推进

在任务执行和等待过程中，Assistant 不等用户发问，而是在适当时机主动输出：

- 哪些任务逾期
- 哪些材料缺失
- 哪些风险有变化
- 哪些待办应优先处理

#### F. 预处理与半自动承接

在规则清晰场景下，Assistant 可先完成：

- 自动预审
- 自动规则核验
- 自动补件单生成
- 自动工单预分类

然后由人工确认、接管或继续推进。

#### G. 长尾托底

当用户提出的需求无法高置信命中标准任务模板时，系统不应直接失败，而应进入开放任务缓冲层。

典型情况包括：

- 低频小众任务
- 新政策带来的专项任务
- 原本不在系统里的工作语义型任务
- 跨模板的临时组合型需求

此时 Assistant 仍可以：

- 先归纳任务意图
- 提供近似模板候选
- 降级为通用任务协助
- 记录为长尾任务样本

#### H. 统一交付

系统最终应将复杂任务的中间结果、子任务结果和待确认动作统一汇总，输出一个可消费、可追溯、可继续推进的交付结果。

### 用户价值

- 更贴近真实工作表达，而不是强迫用户先拆任务
- 减少跨系统查找、切换和手工整理成本
- 节省高频重复劳动，提高输出一致性
- 减少遗漏，加快流转和任务推进
- 让系统先承担低附加值劳动，把人工时间集中在异常和判断环节
- 减少“匹配不到就无法使用”的断裂体验
- 让 Assistant 从“工具”升级为“工作代理”

## 四、核心功能

## 1. 多触点工作入口

`[OpenClaw 借鉴点]`  
`[银行化改造]`

产品不应只有单一聊天入口，而应在多个正式工作触点中存在，包括：

- 岗位工作台 / 业务系统内嵌入口
- 企业 IM
- 移动工作入口
- 文档 / 邮件 / 会议场景入口

其中：

- 主入口必须是 `Session Workspace`，但不能退化成纯聊天框
- 其他入口主要承担触发、提醒、查看摘要和轻量协作

这部分借鉴的是 OpenClaw “跨触点存在”的核心体验，但在银行场景中只限于受控的企业工作触点，而不是开放式个人通信面。

## 2. 工作身份绑定

`[银行化改造]`

每个用户的 Assistant 必须绑定：

- 员工身份
- 岗位
- 部门
- 权限域
- 审批链关系

### 目标

让 Assistant 成为工作身份的延伸，而不是自然人私人助手。

## 3. 会话工作台与任务运行台

`[OpenClaw 借鉴点]`  
`[银行化改造]`

系统不应做成“纯任务入口”，也不应退化成“纯对话入口”。

更合理的产品模型是：

- `Session`：用户与 AI 的工作会话容器，是自然交互的主入口
- `Task`：AI 从会话中识别出的结构化工作单元，是执行、治理、审计和自动化的主对象
- 默认每个 `Session` 都按复杂任务处理，在正式执行前先进入任务规划阶段
- 一个 `Session` 可以关联一个或多个 `Task`
- AI 应持续对 `Session` 做任务识别、重识别和补识别，而不是要求用户一开始手工选定任务类型

### 3.1 Session Workspace

负责统一展示和管理用户当前正在推进的工作会话。

`Session Workspace` 应显示：

- 当前会话摘要
- 当前关注的业务对象
- AI 识别出的任务候选
- 已关联的任务实例
- 当前整理结果
- 推荐下一步动作
- 相关历史会话 / 相似会话

### 3.2 Task Runtime

负责统一展示和管理：

- 标准任务
- 事件触发任务
- 用户主动创建任务
- 长尾任务
- 复合任务

`Task Runtime` 负责承载任务型 UI，而不是由会话消息本身承载执行状态。  
任务运行台应显示：

- 当前任务摘要
- 任务状态
- 当前任务计划
- 关键材料
- AI 整理结果
- 推荐动作
- 历史轨迹

针对默认按复杂任务处理的会话，任务运行台还应显示：

- 子任务列表
- 当前执行计划
- 依赖关系 / 执行进度
- 中间结果汇总

### 3.3 跨 Session 关联发现

AI 不仅要识别“当前会话属于什么任务”，还要识别“当前会话与哪些历史会话有关”。

系统应支持基于以下信号自动发现跨 `Session` 关联：

- 相同客户 / 企业 / 账户 / 案件 / 工单
- 相同材料包、审批件、告警事件或业务单据
- 同一岗位流程中的前后手处理关系
- 同一时间窗口内围绕同一主题的多次交互
- 文本语义上高度相似的处理意图或风险点

关联结果不一定意味着自动合并，而应优先表现为：

- 推荐阅读的相关会话
- 可能延续的历史任务
- 可能重复创建的工作项提醒
- 可供复用的历史处理结论和证据链

对于高置信度场景，系统可以自动建立 `Session Link`；  
对于中低置信度场景，应以候选关系的方式展示给用户确认。

## 4. 任务模板库与任务识别

`[银行化改造]`

系统不应让 AI 自由发明任务类型，而应采用：

- `Session`：工作会话容器
- `Session Router`：会话路由器
- `Task Template`：标准任务模板
- `Task Detector`：任务识别器
- `Task Instance`：任务实例

### 4.1 标准任务模板

每个模板至少定义：

- 任务名称
- 适用岗位
- 触发条件
- 必要上下文
- 可调用技能
- 标准输出结构
- 推荐动作
- 风险等级
- 自动化级别
- 人工接管条件

### 4.2 任务识别

系统应根据：

- 用户输入
- 当前 Session 内的历史消息
- 当前页面上下文
- 当前业务对象
- 最新业务事件
- 岗位身份
- 历史任务轨迹
- 历史 Session 关联图谱

动态判断：

- 当前 Session 属于哪个任务模板或任务组合
- 是否需要新建任务实例，还是继续挂接已有任务
- 当前输入是在延续已有任务，还是开启了新的任务分支
- 当前任务状态是什么
- 是否存在应被自动关联的历史 Session

### 4.3 任务识别置信度机制

建议支持：

- 高置信度：直接创建或关联任务，并自动补挂相关 Session
- 中置信度：提供 2-3 个候选任务 / 关联会话并轻量确认
- 低置信度：降级为澄清问答或通用任务模式

## 5. 复杂任务规划与图执行

`[银行化改造]`

默认情况下，系统不应假设用户输入只对应一个单点动作，而应将每个 `Session` 先视为待规划的复杂任务。

因此在正式执行前，必须先经过任务规划阶段，明确：

- 目标结果是什么
- 需要拆成哪些子任务
- 哪些步骤可并行，哪些步骤有先后依赖
- 哪些步骤可以自动执行，哪些步骤必须人工确认
- 最终以什么结构统一交付

建议将以下能力独立建设为任务规划与图执行层：

- `Task Planner`：任务规划器，负责基于大语言模型从当前 `Session` 中抽取目标、拆解子任务、生成执行计划
- `Task Execution Storage Service`：任务执行存储服务，负责持久化执行过程、节点结果、中间状态、失败原因、回退结果和最终交付物

这里需要明确一个产品约束：

- 这组 `Plan` 能力的规划、解释、重规划、动态调度与汇总逻辑统一由大语言模型实现
- 员工可以直接在 `Session` 中查看 `Plan`，并按风险策略决定是否审批
- `Plan` 的正式执行由基于大语言模型构建的 `Coordinator` 负责推进，而不是依赖一套刚性硬编码流程
- 原本可拆成 `Task Graph Runtime`、`Task State Machine`、`Task Aggregator`、`Replan Manager` 的能力，不再单独建设为后端服务，而由 `Coordinator + Storage` 组合承接

该层应与 `Session Router`、`Task Detector` 紧密联动：

- 先识别会话意图和任务候选
- 再生成任务计划与执行结构
- 然后进入技能调用和工具执行
- 最后把执行结果与交付物回写 `Session` / `Task` 状态

### 5.1 计划确认

复杂任务在进入正式执行前，应先向用户展示任务计划，而不是直接隐式执行。
计划应直接呈现在当前 `Session` 中，使员工能在同一工作上下文里完成查看、审批、修改建议和继续执行。

计划展示至少应包括：

- 系统理解的任务目标
- 拆解出的子任务列表
- 关键依赖关系和执行顺序
- 预计自动执行的步骤
- 需要人工确认的步骤
- 最终交付物结构

对于不同风险等级的任务，建议采用分级确认机制：

- 低风险：允许默认按计划执行，用户可快速修改
- 中风险：要求用户确认关键步骤后再执行
- 高风险：要求用户显式审批整个计划或关键节点后再执行

不论是否要求显式审批，系统都应保留：

- 当前生效的 `Plan Version`
- 用户在 `Session` 中的审批 / 修改 / 驳回记录
- `Coordinator` 实际按哪个版本推进执行

### 5.2 执行中重规划

复杂任务在执行过程中，不应假设初始计划始终有效，而应支持 `Replan` 机制。

以下情况应触发重规划：

- 新材料、新指令或新业务事件进入
- 原定路径执行失败
- 某个关键前置条件不成立
- 用户中途改变目标或优先级
- 系统识别出更优执行路径

重规划后，系统应明确展示：

- 哪些节点保持不变
- 哪些节点被新增、删除或重排
- 为什么需要调整计划
- 是否需要重新人工确认

重规划不应覆盖原始计划，而应保留计划版本与变更轨迹，以便审计和回放。
重规划后的继续执行，也应仍由同一个 `LLM Coordinator` 基于最新生效 `Plan Version` 推进。
所有执行过程、中间状态和最终结果，都应由 `Task Execution Storage Service` 统一持久化。

## 6. 技能动作面板

`[OpenClaw 借鉴点]`  
`[银行化改造]`

提供岗位动作型入口，例如：

- `缺件检查`
- `规则核验`
- `风险摘要`
- `意见草稿`
- `标准回复`
- `补件单生成`

技能应以岗位动作组织，而不是以通用工具组织。

## 7. 企业知识调用

接入：

- 制度库
- 案例库
- 产品规则库
- 模板库
- 业务 FAQ

并支持基于当前 `Session` 上下文和当前 `Task` 上下文做检索和引用。

## 8. 受控工具调用与流程编排

`[OpenClaw 借鉴点]`  
`[银行化改造]`

通过 Tool Gateway 调用：

- 内部系统查询接口
- 流程发起接口
- 文档生成接口
- 表单预填接口

所有调用必须带权限验证和审计。

系统遵循一个核心原则：

**AI 负责理解和判断准备，流程系统负责路由和执行。**

因此建议引入：

- 会话路由引擎：会话归类、任务识别、跨 Session 关联发现
- 认知引擎：理解、提取、归纳、比对、建议
- 编排引擎：状态转换、路由、工具调用、升级、回退

## 9. 执行轨迹与证据链展示

`[银行化改造]`

对每次输出显示：

- 使用了哪些资料
- 依据了哪些规则
- 调用了哪些技能
- 人工做了哪些修改

针对任务执行过程，还应记录：

- AI 做了什么
- 人工确认了什么
- 哪一步转人工
- 哪一步回退
- 哪一步继续自动化

## 10. 主动触达与摘要输出

`[OpenClaw 借鉴点]`  
`[银行化改造]`

这一部分描述的是主动能力在产品层的主要输出形态，而不是第七章所定义的主动决策机制本身。

建议优先提供：

- 待办提醒
- 风险变化提醒
- 补件提醒
- 今日摘要 / 周摘要
- 优先级建议

这些输出应能够：

- 回挂到相关 `Task`、`Task Group` 或 `Session`
- 区分事件触发、周期触发和会话派生触发
- 支持用户确认、忽略、延后和转为正式任务

## 11. 长尾任务缓冲层

当系统未命中标准任务模板时，不应直接失败，而应进入开放任务缓冲层。

该层负责：

- 任务意图归纳
- 候选模板近似匹配
- 通用任务协助降级
- 长尾任务样本记录

同时应提供一套 `Task Mining & Promotion` 机制，用于：

- 分析高频长尾任务
- 识别值得模板化的新任务
- 将其晋升为标准任务模板

## 12. 人工确认与接管

`[银行化改造]`

在半自动和自动场景中，必须支持：

- 确认后执行
- 中断
- 回退
- 升级给人工
- 查看 AI 已完成内容

其中：

- 人工确认和计划审批应尽量在当前 `Session` 中完成，而不是跳出到割裂的外部流程页
- 人工接管、审批结果、回退结果和恢复执行决策都应被记录为 `Plan` / `Task` 的正式轨迹
- 执行轨迹的权威存储应通过 `Task Execution Storage Service` 完成，而不是仅保留在协调器运行态中

## 五、权限设计

这是银行版 PRD 的核心约束之一。

## 1. 权限原则

- 最小权限原则
- 动作分级原则
- 按岗位、Session 关联对象和任务上下文继承权限
- 不允许 Assistant 持有全局超级权限

## 2. 权限分层

### A. 数据读取权限

决定 AI 可以读取哪些系统、哪些字段、哪些业务对象的数据。

### B. 内容生成权限

决定 AI 可以生成哪些类型的草稿、建议、报告、提醒。

### C. 流程发起权限

决定 AI 是否可以发起补件、待审、转派等动作。

### D. 自动执行权限

只有在规则清晰、风险可控场景下，允许阈值内自动执行。

## 3. 权限控制方式

建议采用：

- `RBAC`：按岗位和角色控制
- `ABAC`：按 Session / Task 上下文、客户域、案件状态、组织层级细化控制

## 4. 权限审计要求

每次读取和执行必须记录：

- 谁发起
- 用了什么权限
- 访问了什么数据
- 触发了什么动作

## 七、主动能力的决策与受控执行

银行版 Assistant 的主动能力，不应被理解为“AI 自由想到什么就做什么”，而应被定义为：

**Assistant 基于 `Session Memory`、`Task Memory`、业务事件、周期调度和治理策略，对后续工作机会进行持续判断，并在权限允许范围内生成提醒、建议、候选任务、预处理结果或受控执行动作的能力。**

`[OpenClaw 借鉴点]`  
`[银行化改造]`

因此，本章的核心不是“主动发了什么”，而是要明确：

- Assistant 看到了什么
- Assistant 如何判断是否需要主动介入
- Assistant 可以主动到什么程度
- 主动行为如何回挂到任务系统

### 1. 主动能力的本质

主动能力的本质，是一种持续运行的受控决策机制。

它不是独立于 `Session` 和 `Task` 的附加功能，而是建立在以下基础之上：

- `Session Memory`：当前会话目标、近期消息、相关历史会话、候选任务
- `Task Memory`：当前任务状态、执行计划、子任务状态、待补充信息
- 业务事件流：材料更新、状态变化、风险变化、外部系统回传
- 周期调度：每天、每周、固定时间窗口触发的例行判断
- 治理约束：权限、风险等级、岗位边界、确认规则、审计要求

只有在这些输入的共同作用下，Assistant 才能进行“是否需要主动做点什么”的判断。

### 2. 主动能力的触发机制

建议至少支持三类触发：

其中，A / B 属于系统主动触发，C 属于用户发起后的派生触发。

#### A. 事件触发

这是一类真正的系统主动触发。

适用于：

- 某个任务状态发生变化
- 某份材料补齐或失效
- 某个风险事件、告警事件或业务回执进入系统

这类触发不要求用户此刻正在发起会话，而是系统基于最新业务变化主动判断是否需要介入。

#### B. 周期 / 定时触发

这也是一类真正的系统主动触发。

适用于：

- 每天早上生成今日优先任务建议
- 每日固定时间巡检逾期件、缺件件和高风险件
- 每周五 17:00 生成周收口清单和下周跟进建议
- 每周固定时间为某岗位生成案件排布建议

这类触发强调在固定节奏下对 `Session`、`Task` 和业务状态重新扫描、重新排序和重新判断。

#### C. 会话触发

这类触发不属于“纯系统主动”，而属于“用户发起后的派生触发”。

适用于：

- 用户刚完成一次 `Session`
- 用户在当前会话中表达了新的目标变化
- 会话上下文足以推断出下一步工作机会

它的本质不是系统无中生有地发起动作，而是用户先打开了会话入口，Assistant 再基于当前上下文做延伸判断、补充建议或后续动作生成。

### 3. 主动能力的决策输入

Assistant 在决定是否主动介入时，至少应综合以下信息：

- 当前 `Session Memory`
- 当前 `Task Memory`
- 相关历史 `Session` 与跨 `Session` 关联
- 当前任务计划和任务图执行状态
- 外部事件和最新业务数据
- 岗位权限与风险边界
- 周期性调度策略

因此，主动能力本质上不是“AI 自我发挥”，而是“基于 Memory 的上下文判断 + 基于策略的受控决策”。

### 4. 主动能力的决策输出

在做出主动判断后，系统可以产生以下几类输出：

从触发来源看，A / B 更偏向系统主动输出，C 更偏向用户发起后的派生输出；  
但三者并不是完全割裂的，而是共用同一套决策框架。

#### A. 提醒

适用于：

- 待办提醒
- 逾期提醒
- 补件提醒
- 风险变化提醒

这类输出最典型地来自 A / B，即事件触发和周期触发。

#### B. 建议

适用于：

- 今日优先任务建议
- 案件排布建议
- 跟进建议
- 风险优先级建议
- 每日任务清单
- 每周收口建议

这类输出既可以来自 A / B，也可以来自 C。  
其中，跨任务排序、每日清单、每周收口更偏系统主动；围绕当前会话的下一步建议更偏用户发起后的派生判断。

#### C. 预处理结果

适用于：

- 自动生成任务 briefing
- 自动整理背景材料
- 自动起草标准化输出
- 自动生成待确认动作

这类输出通常更偏 C，即用户发起会话后，由 Assistant 顺着当前上下文继续向前准备。

#### D. 候选任务 / 子任务

适用于：

- 自动识别新的后续动作
- 自动发现新的补充核验事项
- 自动生成待确认任务或候选子任务

这类输出既可能来自 A / B，也可能来自 C。  
当外部事件或周期巡检发现新的工作机会时，它更偏系统主动；当用户会话推进到某一步自然暴露出后续动作时，它更偏派生触发。

#### E. 受控执行动作

仅适用于低风险、规则明确且权限允许的场景，例如：

- 自动整理任务清单
- 自动刷新优先级排序
- 自动准备待确认材料包

这类输出在银行场景中应主要用于 A / B，即系统真正主动触发的低风险场景；  
对于 C，更适合先形成待确认动作，再由用户决定是否执行。

### 5. 主动能力与任务系统的关系

主动能力的输出不能停留在消息层，而必须回挂到任务系统。

建议支持以下几种落点：

- 更新已有 `Task`：刷新状态、优先级、待办或风险标记
- 生成候选 `Task`：等待用户确认后进入正式任务流
- 更新 `Task Group`：形成跨任务的排序、收口和调度建议
- 附着为执行辅助产物：作为 briefing、摘要、草稿、待确认动作被任务引用

也就是说，主动能力不是任务系统外的一层“贴片提醒”，而是任务系统内部的持续判断与激活机制。

### 6. 周期性主动能力设计

周期性主动能力是银行场景中特别重要的一类，因为很多工作并不是由单一事件驱动，而是由固定节奏驱动。

它的核心不是“按时发提醒”，而是：

**按固定节奏重新扫描 Memory、Task 和业务状态，并生成一批可进入任务系统的受控判断结果。**

因此建议引入 `Proactive Schedule Policy`，至少定义：

- 适用对象：个人 / 岗位 / 团队 / 任务组
- 触发频率：每日 / 每周 / 指定工作日 / 指定时间
- 扫描范围：哪些 `Task`、哪些 `Session`、哪些业务对象
- 输出类型：提醒 / 建议 / 预处理 / 候选任务
- 风险级别与是否需要确认

周期性主动能力生成后，优先落为以下几种形态：

- 任务组合级建议清单
- 某个任务组的优先级重排
- 一批待确认的候选任务
- 周期性 briefing / summary

系统还应控制两个风险：

- 不能把周期性主动做成高频噪音推送
- 不能让周期性结果脱离任务系统单独漂浮

### 7. 主动能力的执行边界

建议将主动能力默认分成三档：

- 只读型：只提醒、只建议、只展示
- 确认型：先生成动作建议，待用户确认后执行
- 受控执行型：在低风险场景下自动执行有限动作

在银行场景里，默认应以“只读型”和“确认型”为主，“受控执行型”只在边界清晰的场景中逐步放开。

### 8. 当前不建议优先做的主动能力

- 高自治外部执行
- 未经确认直接触发高风险动作
- 脱离任务系统自由生成生产技能

## 八、升级路径

这个产品不能只按功能版本理解，而应按“岗位劳动承接成熟度”理解。

建议按六级演进。

## Level 0：只读解释

能力：

- 问答
- 检索
- 总结
- 解释

产品形态：

- 仍以单次查询为主
- 尚未形成稳定的 `Session` / `Task` 工作模型

价值：

- 初步替代信息查询劳动

## Level 1：Session 理解与任务路由

能力：

- `Session Workspace`
- `Session Router`
- 任务识别与任务实例化
- 会话上下文组织
- 跨 `Session` 关联发现

产品形态：

- AI 从聊天入口升级为工作会话入口
- 能把自然表达持续路由到一个或多个 `Task`

价值：

- 建立从 `Session` 到 `Task` 的工作主线
- 为后续规划、主动能力和执行能力提供底层对象模型

## Level 2：复杂任务规划与结构化辅助

能力：

- `Task Planner`
- 任务拆解
- 执行计划生成
- 计划确认
- 结构化建议与预处理结果

产品形态：

- 默认将用户 `Session` 视为复杂任务
- 在执行前先规划，而不是直接执行动作

价值：

- 替代大量认知整理和执行准备劳动
- 让 Assistant 从“回答问题”升级为“组织工作”

## Level 3：主动判断与任务激活

能力：

- 基于 `Session Memory` 与 `Task Memory` 的持续判断
- 事件触发主动能力
- 周期 / 定时触发主动能力
- 提醒、建议、候选任务生成
- `Task Group` 级排序与收口建议

产品形态：

- 系统开始在用户不开口时主动介入
- 会围绕任务状态、业务事件和固定节奏持续产出受控主动结果

价值：

- 开始替代“记得做、记得跟、记得排优先级”这类工作管理劳动
- 让 Assistant 从被动工具升级为持续运行的工作体

## Level 4：人工在环的受控执行

能力：

- `LLM Coordinator` 驱动的动态执行
- `Task Execution Storage Service`
- Tool Gateway 调用
- AI 先处理、人工确认
- 回退、重试、升级人工
- 完整执行轨迹和证据链

产品形态：

- 复杂任务可以按计划被分步执行
- 默认以“确认型执行”为主，而不是全自动执行

价值：

- 开始显著替代岗位任务片段
- 将执行能力纳入治理和审计框架

## Level 5：阈值内自动执行与数字岗位承接

能力：

- 在规则和阈值允许范围内自动完成标准化动作
- 异常分流与人工兜底
- 周期性主动与自动执行联动
- 以数字岗位方式稳定承接部分岗位劳动

产品形态：

- 在清晰边界内形成稳定机器产能
- 部分岗位劳动由数字岗位持续承接

价值：

- 为 FTE 替代、组织压降和岗位重构提供基础

## 十、成功指标

建议把指标同时分成两种视角：

- 一种是指标层次：看产品是否被使用、是否有效、是否可控、是否产生替代价值
- 一种是优先级：看哪些指标应作为当前阶段的核心目标，哪些指标更适合作为中后期结果指标

## 1. P0：核心成败指标

这些指标直接决定这条产品路线是否成立，应作为当前阶段最重要的成功判断。

### A. Session 与任务路由

- `Session -> Task` 路由成功率：成功从 `Session` 路由到至少一个有效 `Task` 的会话占比
- 单 `Session` 多任务识别率：被正确识别为包含多个任务或任务组合的 `Session` 占比
- 跨 `Session` 关联准确率：系统自动建立或推荐的 `Session Link` 中，被人工验证为正确关联的占比
- 候选任务挂接成功率：被识别出的候选任务中，最终成功挂接到正式任务流的占比

### B. 任务规划与执行准备

- 计划确认率：生成的任务计划中，被用户确认进入执行阶段的占比
- 计划一次通过率：无需修改或补充说明就被直接确认的任务计划占比
- 重规划率：执行过程中触发 `Replan` 的任务占比
- 任务拆解准确率：任务规划器拆出的子任务结构与人工标注或复核结果一致的占比

### C. 受控主动能力

- 事件触发主动结果采纳率：由事件触发产生的提醒、建议或候选任务中，被用户接受、执行或确认的占比
- 周期性主动结果采纳率：由定时调度产生的主动结果中，被用户接受、执行或确认的占比
- 主动提醒噪音率：被用户忽略、关闭、标记为无用或重复的主动提醒占比
- 候选任务转正率：系统生成的候选任务中，最终被确认并转化为正式 `Task` 的占比

### D. 执行质量与治理安全

- 任务识别准确率：系统对任务模板、任务类型和任务归属的识别结果与人工复核一致的占比
- 错误率：输出结果或执行结果中，被人工确认存在事实、规则或流程错误的占比
- 人工接管率：进入执行阶段后，需要人工中途接管的任务占比
- 人工接管后推翻率：人工接管后，对 AI 原判断、原计划或原执行结果进行实质性推翻的占比
- 策略违规率：主动或执行行为中，违反权限、风控或流程约束的占比
- 证据链完整率：可完整回放数据来源、规则依据、执行步骤与人工修改记录的任务占比

## 十一、总结

如果把这份 PRD 压缩成一句话，那么银行版 OpenClaw 的产品目标不是：

**给每个员工一个聪明聊天助手**

而是：

**给每个员工一个绑定工作身份、以 `Session` 为工作入口、以 `Task` 为治理与执行单元、具备任务规划、主动能力和受控执行能力，并可向数字岗位演进的 Assistant OS。**

这也意味着，这个产品的价值并不是一次性给出更好的回答，而是持续承接银行员工的真实工作过程。

从建设路径看：

- 第一阶段卖的是岗位生产力
- 第二阶段卖的是可治理、可规划、可审计的工作能力
- 第三阶段卖的是机器产能和数字岗位承接能力

因此，这套产品的底层不应只是“问答 + 技能调用”，而必须同时具备：

- `Session Workspace` 与 `Session Router`
- 任务模板体系、任务识别与任务实例化
- 复杂任务规划与任务图执行能力
- 基于 `Session Memory` 与 `Task Memory` 的主动判断能力
- 长尾任务缓冲与任务晋升机制
- 人工确认、回退、升级人工与执行轨迹能力
- 可度量的效率、治理和 `FTE` 映射能力

只有这样，它才是一个真正适合银行场景、也能支撑组织级变化的产品形态。
