# 银行 AI Agent / AI Assistant 方案稿

## 一、问题定义

面向银行设计 AI 产品时，一个常见分歧是：我们究竟应该主打“AI Agent 员工”，还是应该主打“给员工配置 AI Assistant Agent”。

如果只从概念上看，前者听起来更先进，后者听起来更像过渡方案；但如果结合银行的业务特点、治理要求和采购逻辑，这个问题的答案并不是谁更前沿，而是谁更符合银行的真实落地路径。

本方案的核心观点是：

**银行短中期应以 AI Assistant Agent 为主形态，以提升现有岗位能力为第一目标；同时底层平台需按 Agent-native 架构设计，为少数标准化场景逐步演进出真正的 AI Agent 员工预留空间。**

简化表达就是：

**Assistant-first，Agent-native。**

---

## 二、结论先行

### 1. 银行当前的主需求

银行当前更需要的是：

- 给客户经理配置 AI Assistant
- 给审批岗配置 AI Assistant
- 给风控岗配置 AI Assistant
- 给合规岗配置 AI Assistant
- 给运营岗配置 AI Assistant

即，优先通过 Assistant 形态增强现有岗位的认知效率、材料处理效率和标准输出能力。

### 2. 银行中长期的演进方向

在少数场景上，银行会逐步接受真正的 AI Agent 员工，但通常仅限于：

- 高频
- 稳定
- 输入输出标准化
- 风险边界清晰
- 可审计
- 可人工接管

这些数字岗位首先会出现在预审、初筛、标准运营核查、材料校验、流程跟进等场景，而不会首先出现在关键责任决策岗位。

### 3. 核心判断

银行不是不需要 AI 员工，而是：

- 当前更需要 `有 AI 助手的员工`
- 长期才会形成 `少数可治理的 AI 员工岗位`

---

## 三、为什么银行应优先选择 AI Assistant Agent

### 1. 责任主体更清晰

银行大量核心岗位承担的不是简单劳动，而是判断、签字、解释和责任。Assistant 形态不会改变责任归属，只会增强责任主体的认知能力，因此更容易纳入现有管理体系。

### 2. 权限体系更容易继承

银行的权限高度分层、分条线、分系统。Assistant 可以继承员工本人现有权限边界，避免重新设计复杂的“数字岗位身份和授权模型”。

### 3. 合规和内控更容易通过

银行更容易接受“AI 作为辅助工具，由授权人员最终确认”的模式，而不容易一开始接受“AI 独立执行关键业务动作”的模式。

### 4. ROI 更容易证明

Assistant 类场景的价值容易被量化，例如：

- 查询效率提升
- 材料整理时间下降
- 文稿撰写时间下降
- 初步检查效率提高
- 输出一致性增强

### 5. 推广阻力更小

Assistant 是岗位增强，AI 员工是岗位重构。银行更愿意先采购生产力升级，而不是一开始推动组织变革。

---

## 四、为什么不能一开始直接做“AI 员工”

### 1. 责任链难以闭合

如果 AI 独立完成关键动作，一旦结果出错，责任归属会变得复杂。银行不会轻易接受责任不清的系统直接进入关键流程。

### 2. 授权模型复杂度极高

给 AI 员工配置权限，不是增加一个账号，而是设计一个组织主体：

- 属于哪个部门
- 接受谁管理
- 拥有什么权限
- 是否满足岗位分离原则
- 策略变更由谁审批

### 3. 监管叙事成本高

“AI 辅助人”容易解释，“AI 替代岗位执行关键动作”难以解释。前者是升级工具，后者是改变治理结构。

### 4. 组织接受度有限

业务、合规、科技、内控、审计、采购多方共同决策时，Assistant 更容易达成共识；AI 员工更容易引发对风险、替代、治理和职责边界的争议。

---

## 五、如何区分 AI 功能、AI Assistant 和 AI Agent 员工

### 1. AI 功能

指系统中的单点智能能力，例如：

- 摘要
- 分类
- 检索
- 生成回复
- 风险评分

这是能力组件，不是员工。

### 2. AI Assistant

指围绕某个员工工作的智能助手，特点包括：

- 知道用户是谁
- 知道权限边界
- 帮助组织信息和生成建议
- 可以调用工具
- 默认不独立承担结果

本质是：**人主导，AI 辅助。**

### 3. AI Agent 员工

指被组织视为数字岗位的 Agent，特点包括：

- 有明确岗位定义
- 有独立任务池
- 有职责边界
- 有动作权限
- 有输出标准
- 有升级机制
- 有质检和绩效统计

本质是：**在明确边界内承担稳定产能。**

---

## 六、银行岗位可拆分为三层

为了判断哪些场景适合 Assistant，哪些场景适合 Agent 员工，可以把银行岗位拆成三层。

### 1. 认知辅助层

典型任务：

- 信息获取
- 材料理解
- 知识检索
- 文本生成
- 规则提示
- 案例比对

适合形态：`AI Assistant`

### 2. 流程执行层

典型任务：

- 收单
- 校验
- 流转
- 填报
- 归档
- 分发
- 触发下游动作

适合形态：`局部 Agent 化`

### 3. 责任决策层

典型任务：

- 授信批准
- 风险定级
- 合规定性
- 客户建议
- 异常处置确认

适合形态：`以人工为主，AI 只做辅助`

结论是：

**银行大多数岗位应当认知辅助层全面 Assistant 化，流程执行层局部 Agent 化，责任决策层长期保持人工主导。**

---

## 七、推荐的产品路线

### 第一阶段：岗位增强

目标：让现有岗位的人更强、更快、更稳。

重点场景：

- 客户经理助手
- 审批助手
- 风控助手
- 合规助手
- 运营助手

核心能力：

- 统一知识问答
- 会前准备
- 材料摘要
- 缺件识别
- 风险提示
- 草稿生成
- 跨系统信息整合

成功标志：

- 日活/周活可观
- 单任务耗时显著下降
- 输出质量和一致性提升

### 第二阶段：能力沉淀

目标：把高频任务沉淀为可治理的自动化能力。

重点变化：

- 从“人发起、AI 协助”
- 变成“系统自动接管部分任务片段”

典型能力：

- 材料完整性自动检查
- 补件清单自动生成
- 标准案例初步匹配
- 运营工单自动分类与预处理
- 交易告警自动聚合与预分析

平台要求：

- 任务编排
- 工具调用
- 权限控制
- 流程状态管理
- 审计留痕
- 异常升级和人工接管

### 第三阶段：数字岗位化

目标：将部分成熟能力制度化为银行认可的数字岗位。

候选岗位：

- KYC 预审专员 Agent
- 贷前资料初审 Agent
- 交易告警初筛 Agent
- 贷后预警跟踪 Agent
- 标准运营核查 Agent

成功标志：

- 有稳定任务池
- 有明确职责边界
- 有质量考核指标
- 有正式进入业务链条的资格

---

## 八、平台架构建议

面向银行的产品架构，不应只做一个聊天入口，而应采用“前台 Assistant + 中台 Agent 能力 + 底层治理底座”的结构。

### 1. 前台：岗位 Assistant 层

面向对象：

- 客户经理
- 审批岗
- 风控岗
- 合规岗
- 运营岗

交互特征：

- 对话式工作入口
- 任务式协作入口
- 嵌入现有工作台
- 与日常流程结合

### 2. 中台：Agent 能力层

能力包括：

- 任务拆解
- 工具编排
- 多系统调用
- 状态跟踪
- 规则/阈值控制
- 自动执行和转人工机制

### 3. 底座：治理和集成层

能力包括：

- 身份与权限体系
- 数据访问控制
- 知识治理
- 日志与审计
- 模型评测
- 风险控制
- 异常处理
- 人机接管

---

## 九、与传统银行 IT 的本质区别

这套方案并不是“传统银行 IT 加一个 AI 功能”。

传统银行 IT 主要解决的是：

- 流程固化
- 交易记录
- 规则执行
- 权限控制
- 状态流转

而银行 AI 体系开始解决的是：

- 材料理解
- 知识检索
- 证据归纳
- 建议生成
- 多步骤任务协同
- 局部自主执行

因此，两者最核心的区别在于：

**传统银行 IT 的对象是流程和交易，AI 体系的对象开始延伸到认知和执行。**

这意味着系统角色发生了变化：

- 传统 IT 更像工具、账本、流程容器
- AI 体系更像助手、分析者、建议者、局部代理执行者

但两者不是替代关系，而是分层关系：

- 传统 IT 提供数据、接口、权限、流程、日志底座
- AI 体系叠加在其上，提供智能理解、建议和自动化能力

---

## 十、面向银行客户的建议话术

如果用于售前或客户沟通，建议不要一开始直接强调“AI 员工替代人”，而应采用以下表述逻辑：

### 1. 第一层表达：岗位增强

“我们首先帮助现有岗位更强、更快、更稳，让员工先真正用起来。”

### 2. 第二层表达：流程能力化

“在高频、稳定任务上，我们把个人使用能力沉淀成组织可治理的自动化能力。”

### 3. 第三层表达：数字岗位化

“在少数成熟场景，我们进一步将能力制度化为受控的数字岗位，而不是一开始追求全能 AI 员工。”

这种表达方式更符合银行的心理预期，也更容易通过业务与治理双重审视。

---

## 十一、最终结论

如果必须给出一句最简洁的判断，那么就是：

**银行首先需要的是有 AI 助手的员工，而不是替代员工的 AI。**

更完整的表达是：

- 短中期主形态：`AI Assistant Agent`
- 中长期终局之一：`少数可治理的 AI Agent 员工`
- 推荐路线：`Assistant-first，Agent-native`

因此，面向银行做 AI，不应把重点放在“我们能造多少 AI 员工”上，而应把重点放在三件事上：

1. 先让现有岗位的人更强、更快、更稳
2. 再把高频任务沉淀为可治理的组织能力
3. 最后把少数成熟能力制度化为数字岗位

这才是银行场景里最现实、最可落地、也最有长期价值的路线。
