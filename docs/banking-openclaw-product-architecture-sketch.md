# 银行版 OpenClaw 的产品架构草图

## 一、设计目标

如果参考 OpenClaw 的形态去构建银行员工 AI Assistant，那么最重要的不是复制一个“个人 AI 助理”，而是构建一个：

**绑定工作身份、围绕任务运行、以技能承接岗位动作、受治理严格约束、并可向数字岗位演进的 Assistant OS。**

因此，这份架构草图的目标不是描述一个普通聊天机器人，而是描述一个适合银行场景的工作代理系统。

这个系统应同时满足五个目标：

1. 让员工日常高频使用，而不是只在演示时好看
2. 能深度嵌入岗位工作流，而不是悬浮于系统之外
3. 能沉淀高频任务和技能，为 Agent 化做准备
4. 能被权限、审计和风险体系治理
5. 能为后续岗位劳动承接和 FTE 替代测算提供数据基础

---

## 二、总体架构理念

如果用一句话描述银行版 OpenClaw 的总体架构，我会把它定义为：

**Assistant OS + Task Engine + Governed Skill Runtime + Banking Control Plane**

它的核心不是单个聊天入口，而是一层运行在银行正式工作触点之上的“工作代理层”。

从架构上看，可以拆成八层：

1. 交互接入层
2. 身份与权限层
3. 任务上下文层
4. 记忆与知识层
5. 技能与工具层
6. Agent 编排与执行层
7. 治理与审计层
8. 度量与替代分析层

下面逐层展开。

---

## 三、交互接入层

这一层决定员工如何触达 Assistant。银行版不应把聊天作为唯一工作面，而应采用“主工作面 + 辅助触点”的设计。

## 1. 主工作面

主工作面应嵌入正式业务场景，例如：

- 授信审批工作台
- 运营处理工作台
- 风控分析工作台
- 客户经理工作台
- 合规审查工作台

主工作面承担的不是闲聊，而是：

- 展示任务上下文
- 展示 AI 已整理的信息
- 提供岗位动作入口
- 展示建议、草稿、异常点、下一步行动

## 2. 辅助触点

辅助触点可以包括：

- 企业 IM
- 移动端工作入口
- 邮件 / 文档入口
- 会议纪要与知识入口

这些触点的作用是：

- 便捷触发
- 查看摘要
- 接收提醒
- 快速发起任务

但不应成为唯一工作面。

## 3. 接入层的核心原则

- 主入口必须任务化
- 辅助入口必须受控
- 所有入口共享同一任务上下文
- 不允许出现“不同入口不同记忆状态”的割裂体验

---

## 四、身份与权限层

这一层是银行版和个人版 OpenClaw 最根本的区别。

在银行里，Assistant 不是“跟着自然人”的，它必须“跟着工作身份”。

## 1. 身份绑定模型

每个 Assistant 实例都应绑定：

- 员工 ID
- 部门
- 岗位
- 组织层级
- 岗位职责
- 审批链关系
- 可访问系统域

## 2. 权限控制模型

建议同时支持：

- `RBAC`：按角色分配权限
- `ABAC`：按属性和上下文细化权限

例如：

- 同样是审批岗，不同级别能看到的数据不同
- 同样是客户经理，不同条线能访问的客户域不同
- 同样是风控岗，不同案件类型能执行的动作不同

## 3. 会话权限原则

Assistant 的每一次读取、生成、建议、调用工具、发起动作，都必须继承当前身份和当前任务上下文，而不是使用一个全局超级权限。

## 4. 身份层的关键目标

- 保证 Assistant 不越权
- 保证员工换岗、离岗、调岗时，能力边界同步调整
- 为未来数字岗位权限模型预留基础

---

## 五、任务上下文层

这层是整个银行版 Assistant OS 的核心。

如果没有任务上下文，系统最终就会退化成“有记忆的聊天工具”；有了任务上下文，系统才有机会进化成真正的岗位工作代理。

## 1. 任务对象模型

系统应显式定义“任务”作为一等对象。  
任务对象至少包含：

- 任务 ID
- 任务类型
- 所属业务对象（客户 / 案件 / 审批件 / 工单）
- 当前状态
- 优先级
- 所属人 / 协作人
- 已完成步骤
- 待完成步骤
- 输入材料
- 输出结果
- AI 参与轨迹

## 2. 任务状态机

建议对不同任务类型建立状态机，例如：

- 待处理
- AI 已预处理
- 人工复核中
- 待补件
- 已提交
- 异常升级
- 已完成

## 3. 人机协同轨迹

每个任务都应记录：

- AI 做了什么
- 人确认了什么
- 哪一步转人工
- 哪一步返回 AI

这不仅用于体验，更用于后续能力沉淀和 FTE 测算。

## 4. 任务层的关键价值

- 把工作从“对话”转成“任务”
- 把岗位劳动显性化
- 为 Agent 化和数字岗位化打基础

---

## 六、记忆与知识层

这一层不能简单照搬个人助理的自由记忆逻辑，而要建立治理化记忆架构。

建议拆成五层。

## 1. Identity Memory

记录员工工作身份相关信息：

- 岗位
- 部门
- 权限范围
- 常用工作对象
- 组织关系

## 2. Role Memory

记录岗位相关长期上下文：

- SOP
- 常用模板
- 岗位偏好配置
- 岗位标准输出结构

## 3. Task Memory

记录当前任务及其最近上下文：

- 当前案件状态
- 材料变化
- 待处理事项
- AI 与人工已完成内容

## 4. Enterprise Memory

记录组织层知识：

- 制度
- 政策
- 产品规则
- 历史案例
- 标准回复

## 5. Execution Trace

这是银行版里最重要的一层，记录：

- 使用了哪些数据
- 调用了哪些工具
- 做出了哪些判断
- 输出基于哪些依据

这一层必须可追溯、可审计、可回放。

## 6. 记忆层的设计原则

- 记忆必须分层
- 记忆必须受权限约束
- 记忆必须支持生命周期管理
- 记忆不能自由自生长
- 记忆必须服务于任务，而不是服务于“人格化陪伴”

---

## 七、技能与工具层

这一层是银行版 Assistant 的“工作能力层”。

和 OpenClaw 倾向于通用技能不同，银行版技能必须围绕岗位动作组织。

## 1. 技能分类

建议分成四类。

### A. 信息组织类技能

例如：

- 材料摘要
- 客户画像整理
- 风险线索归纳
- 制度依据提取

### B. 规则检查类技能

例如：

- 缺件检查
- 基础规则核验
- 标准格式检查
- 阈值对比

### C. 产出生成类技能

例如：

- 审查意见草稿
- 标准回复草稿
- 补件清单生成
- 工作汇总生成

### D. 流程推进类技能

例如：

- 发起补件流程
- 提交待复核任务
- 触发风险上报
- 分发标准工单

## 2. 工具接入原则

所有工具必须通过统一 Tool Gateway 接入，不允许 Agent 直接自由访问系统。

Tool Gateway 需要提供：

- 身份校验
- 动作分级
- 参数校验
- 调用审计
- 限流与拦截

## 3. 技能版本化

技能必须支持：

- 版本管理
- 配置管理
- 灰度发布
- 回滚
- 审计记录

## 4. 技能层的核心原则

- 强岗位化
- 强 SOP 化
- 强权限化
- 强审计化

---

## 八、Agent 编排与执行层

这是从 Assistant 向 Agent-native 演进的关键层。

## 1. 编排核心

这一层负责：

- 任务拆解
- 技能选择
- 工具调用排序
- 多步骤执行
- 状态跟踪
- 异常升级

## 2. 执行分级

建议把执行能力显式分成五级：

- `L0` 只读解释
- `L1` 结构化建议
- `L2` 草稿与预处理
- `L3` 半自动执行
- `L4` 阈值内自动执行

数字岗位化阶段可以视为：

- `L5` 数字岗位承接

## 3. 人机接管机制

Agent 编排必须内建：

- 人工确认节点
- 人工接管节点
- 异常升级节点
- 任务回退节点

## 4. 编排层的关键价值

- 支撑从“建议”到“承接劳动”的升级
- 避免系统停留在问答层
- 为数字岗位化提供技术基础

---

## 九、治理与审计层

这一层不是附属模块，而是银行版 OpenClaw 的基础设施。

建议至少包含以下模块。

## 1. 权限治理

- RBAC / ABAC
- 数据访问控制
- 动作授权控制
- 最小权限原则

## 2. 风险治理

- 能力分级
- 风险分层
- 黑白名单动作控制
- Prompt 注入防护
- 越权调用防护

## 3. 审计治理

- 数据来源留痕
- 工具调用留痕
- 输出依据留痕
- 人工修改留痕
- 回放与追踪

## 4. 变更治理

- Prompt 版本管理
- Skill 版本管理
- 模型版本管理
- 灰度发布
- 回滚机制

## 5. 运营治理

- 能力监控
- 质量监控
- 异常告警
- 使用分析

---

## 十、度量与替代分析层

如果系统未来要服务于岗位劳动承接和 FTE 替代测算，这一层必须从第一天就设计。

## 1. 任务计量

记录：

- 任务总量
- 任务类型
- 自动处理比例
- 人工接管比例
- 平均处理时长

## 2. 人机分工计量

记录：

- AI 承接步骤
- 人工确认步骤
- 人工主导步骤
- 返工步骤

## 3. 产能计量

记录：

- 自动处理率
- 质量合格率
- SLA 达成率
- 接管率
- 返工率

## 4. FTE 映射

记录：

- 工时节省
- 理论 FTE 替代
- 可实现 FTE 替代
- 可兑现 headcount 压降

这一层的存在，决定了产品能否从“喜欢用”走向“可规划”。

## 十二、最后的判断

如果把这份架构草图压缩成一句话，那么银行版 OpenClaw 的本质不是：

**一个更会聊天的 AI 助理**

而是：

**一个围绕员工工作身份运行、围绕岗位任务组织、围绕治理能力约束、并最终服务于数字产能演进的 Assistant OS。**

它和个人版 OpenClaw 最大的不同，不在于换了银行场景，而在于：

- 从“个人”变成“工作身份”
- 从“聊天”变成“任务”
- 从“自由记忆”变成“治理化记忆”
- 从“通用技能”变成“岗位动作技能”
- 从“好用”变成“可治理、可沉淀、可测算、可替代”

这才是银行场景下真正可落地的产品架构方向。
