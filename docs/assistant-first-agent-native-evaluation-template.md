# Assistant-first, Agent-native 问题评估表模板

## 一、使用说明

这份模板用于把 `Assistant-first, Agent-native` 路线下的关键问题，从“讨论清单”转化为“可评估、可排序、可推进”的问题项。

建议在以下三种场景中使用：

1. 产品方向讨论：帮助团队统一优先级和判断标准
2. 客户方案设计：帮助识别哪些问题需要先内部决定，哪些要靠客户访谈或试点验证
3. 项目推进管理：帮助把问题落实到 Owner、阶段和下一步动作

建议使用方式：

- 先从 `12 个核心问题` 中挑出与当前项目最相关的 6-8 个
- 逐项填写评估表
- 用同一套打分标准对问题排序
- 将高优先级问题转化为具体行动项

---

## 二、建议的评估维度

每个问题建议从以下维度评估。

### 1. 基本信息

- 问题名称
- 问题描述
- 所属类别
- 当前阶段
- Owner

### 2. 问题属性

- 问题类型：战略 / 场景 / 产品 / 平台 / 治理 / 商业
- 难度类型：认知 / 产品 / 技术 / 治理
- 验证方式：内部决策 / 客户访谈 / 试点验证 / 治理沟通

### 3. 优先级评估

建议按 1-5 分打分：

- 战略影响：答错是否会明显带偏路线
- 当前紧迫性：是否现在就必须回答
- 依赖性：是否是多个后续问题的前提
- 不确定性：当前是否高度模糊、团队缺少共识
- 解决成本：回答它需要多少资源和时间

### 4. 额外分析维度

同样建议按 1-5 分打分：

- 价值密度：一旦解决，能释放多大业务价值
- 可验证性：是否能通过访谈或试点快速验证
- 可逆性：判断错了之后是否容易调整
- 外部依赖程度：是否强依赖客户系统、权限和组织环境
- 时间跨度：是当前问题还是中长期问题

### 5. 结论输出

- 当前判断
- 当前假设
- 主要风险
- 下一步动作
- 预计完成时间

---

## 三、优先级划分标准

为了便于排序，建议统一使用以下优先级定义。

### P0：现在不回答，项目无法定义

典型特征：

- 影响产品方向
- 影响试点对象选择
- 影响第一版核心设计

### P1：试点前必须基本明确

典型特征：

- 影响可落地性
- 影响可治理性
- 可以先带着初步假设，但不能完全模糊

### P2：试点中重点验证

典型特征：

- 很难靠内部讨论得出答案
- 需要真实用户行为和数据反馈

### P3：中长期规划问题

典型特征：

- 重要但不紧急
- 不应现在完全定死
- 可以先定义方向和判断框架

---

## 四、难度分类标准

### 1. 认知难度

问题难在战略判断和价值理解，例如：

- 银行为什么愿意为这个产品买单
- 第一批岗位应该选谁

### 2. 产品难度

问题难在把需求转化为高频使用的产品形态，例如：

- Assistant 为什么会被每天使用
- 嵌入哪个工作流最合适

### 3. 技术难度

问题难在底层系统能力和工程实现，例如：

- 是否显式做任务模型
- 是否支持状态管理和人机交接

### 4. 治理难度

问题难在合规、权限、责任和组织接受度，例如：

- 自动执行阈值怎么设
- 哪些场景永远不能自动化

## 七、推荐的实际使用顺序

如果希望把模板用起来，而不是停留在文档层，建议按以下顺序推进：

1. 先填写问题 1-5，完成第一版产品和试点方向定义
2. 再填写问题 6-9，完成平台和治理层设计
3. 最后填写问题 10-12，形成中长期演进判断

最重要的是，不要一开始就平均用力。真正应该先做深的，永远是那些：

- 高战略影响
- 高当前紧迫性
- 高依赖性
