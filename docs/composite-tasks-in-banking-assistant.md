# 银行 AI Assistant 中的复合任务：识别、拆解与编排

## 一、问题定义

在真实银行工作里，员工提出的需求，很多并不是单一任务，而是多个任务类型的组合。

例如：

- “帮我把今天高风险且待补件的客户先排个优先级，再把前 5 个的会前摘要准备出来”
- “先看这批授信件哪些缺关键材料，再把风险高的单独列出来给我”
- “把这几个告警先初筛一下，值得升级的帮我汇总成一份简报”

这类输入的共同特点是：

- 目标导向，而不是模板导向
- 通常包含多个子任务
- 子任务之间存在先后、并行或条件关系
- 最终需要一个统一交付结果

因此，如果银行 AI Assistant 只支持“一个输入匹配一个任务模板”，很快就会显得不够用。

更合理的设计应该是：

**当用户输入涵盖多种任务类型时，系统应将其识别为复合任务，拆解成多个标准子任务，生成任务图，并由 orchestration 引擎按依赖关系执行，最终汇总为一个统一结果。**

一句话概括：

**用户提的是一个需求，系统内部跑的是一个任务组合。**

---

## 二、为什么复合任务是常态，而不是例外

这件事之所以重要，是因为银行员工的真实工作，并不是按任务模板名发生的，而是按工作目标发生的。

员工不会说：

- “请执行任务模板 A，再执行任务模板 B，再执行任务模板 C”

员工更可能说：

- “先把这批客户按优先级排一下，再给我整理重点客户的风险变化”
- “先筛出缺件的，再给高风险件出补件清单”
- “把这个客户的历史交易、风险变化和会议纪要一起整理成会前准备”

也就是说，真实工作天然是：

- 目标化的
- 组合化的
- 跨步骤的

所以在产品设计上，复合任务不应被视为边缘情况，而应被视为：

**任务系统的核心能力之一。**

---

## 三、复合任务的三种典型结构

并不是所有组合型任务都一样。为了后续编排更清楚，我建议至少分成三类。

## 1. 串行组合

一个任务的输出，是下一个任务的输入。

例如：

- 先检查缺件，再生成补件清单
- 先做风险归纳，再生成审查意见草稿
- 先筛客户，再做会前准备

它的结构是：

`Task A -> Task B -> Task C`

这种任务最适合用 pipeline 式编排。

## 2. 并行组合

多个子任务可以同时执行，最后再汇总。

例如：

- 整理客户交易变化
- 整理历史跟进记录
- 整理风险变化
- 最后汇总成会前摘要

它的结构是：

`Task A + Task B + Task C -> Task D`

这种任务适合任务图中的并行节点。

## 3. 条件组合

某个子任务是否执行，取决于前一步结果。

例如：

- 先看客户是否高风险，如果高风险再做会前准备，否则只做简要摘要
- 先检查材料是否齐全，如果缺关键件就生成补件单，否则继续做风险归纳

它的结构是：

`Task A -> if X then Task B else Task C`

这种任务已经非常接近流程编排逻辑。

---

## 四、产品设计上应如何处理复合任务

针对复合任务，我建议采用六步处理链路。

## 第一步：先识别“这是单任务还是复合任务”

系统首先不要急着匹配单一模板，而要先判断：

- 用户输入里是不是有多个动作目标
- 是否包含多个输出要求
- 是否存在明显的“先……再……”“同时……并且……”“如果……则……”结构

### 示例

输入：

- “帮我把今天高风险且待补件的客户先排个优先级，再把前 5 个做会前准备”

系统应先识别：

- 这是一个复合任务，而不是单一任务

这一步的目标是：

**先识别复合性，再决定匹配方式。**

---

## 第二步：做任务拆解

一旦识别出复合任务，系统应将用户需求拆成多个标准子任务。

### 示例

上面的输入可以拆成：

1. `followup_priority_sorting`
2. `customer_meeting_briefing`

进一步补全上下文：

- 范围：今天待跟进客户
- 过滤条件：高风险 + 待补件
- 输出对象：前 5 个客户

### 关键原则

这里不是让 AI 自由发明新任务，而是：

**AI 基于已有任务模板库做组合匹配。**

---

## 第三步：生成任务图

拆解之后，系统不应把这些子任务简单放成一个列表，而应组织成一个任务图。

### 示例一：串行图

```text
[筛选今天待跟进客户]
        ->
[识别高风险且待补件客户]
        ->
[优先级排序]
        ->
[取前5个客户]
        ->
[生成会前准备摘要]
```

### 示例二：并行图

```text
           -> [交易变化整理] -
[客户A] -> [历史跟进整理] --- > [会前摘要汇总]
           -> [风险变化整理] -
```

### 任务图的价值

任务图不是为了好看，而是为了支撑：

- 串行 / 并行 / 条件分支
- 局部重试
- 局部失败回退
- 人工接管
- 结果汇总
- 后续计量分析

---

## 第四步：让用户看到“组合任务计划”

在银行场景中，不建议系统对复合任务默默后台乱跑，特别是在中高风险任务上。

更好的体验是：

### 高置信、低风险场景

系统可直接执行，但同时展示计划，例如：

- 将执行：
  1. 筛选高风险待补件客户
  2. 优先级排序
  3. 生成前 5 个客户会前摘要

### 中置信或高风险场景

系统先做轻量确认，例如：

- “我理解你是想先做客户筛选和优先级排序，再为前 5 个客户生成会前准备，是否继续？”

### 这样做的价值

- 防止误拆解
- 增强可解释性
- 减少用户不信任
- 为高风险场景增加治理缓冲层

---

## 第五步：分步骤执行，而不是一口气黑盒完成

复合任务不应被系统黑盒地“一口气跑到底”。

更合理的方式是：

- 每个子任务独立执行
- 每一步都有中间状态
- 某一步失败时可暂停或回退
- 某一步发现异常时可转人工

### 为什么这一步重要

因为复合任务里经常会发生：

- 某一步信息不足
- 某一步结果改变后续任务范围
- 某一步需人工确认
- 某一步异常导致整个组合任务不能继续

如果没有分步骤执行和可见状态，就很难治理和接管。

---

## 第六步：统一结果汇总

虽然系统内部执行的是多个子任务，但面向用户的交付结果应尽量是统一的。

例如：

- 一个优先级排序列表 + 每个客户的会前摘要
- 一个高风险件清单 + 每件的缺件说明 + 下一步建议
- 一份升级告警简报 + 每条告警的初筛结论

也就是说：

**内部多任务，外部一体化交付。**

这也是为什么需要专门设计 `Task Aggregator`。

---

## 五、从架构角度，支持复合任务意味着什么

如果系统要支持复合任务，架构就不能停留在“任务模板匹配器”这一层，而必须升级成更完整的任务编排体系。

我建议至少增加三个核心能力。

## 1. Task Planner

任务规划器。

负责：

- 判断当前是单任务还是复合任务
- 进行任务拆解
- 识别依赖关系
- 生成任务计划或任务图

它不是完全自由的 agent planner，而应是：

**基于任务模板库的受控规划器。**

## 2. Task Graph Runtime

任务图运行时。

负责：

- 维护多个子任务状态
- 管理串行 / 并行 / 条件分支
- 执行局部重试
- 执行人工接管和回退
- 控制依赖关系

这是复合任务真正运行起来的核心。

## 3. Task Aggregator

任务汇总器。

负责：

- 把多个子任务的结果重新组织成统一交付
- 面向人输出自然语言总结
- 面向系统输出结构化结果

这样用户看到的不是“碎掉的一堆小任务”，而是一个完成后的工作结果。

---

## 六、建议把复合任务分成两个成熟度层级

为了控制复杂度，不建议第一版就支持无限复杂的任务组合。

我建议分两级建设。

## L1：轻量复合任务

特点：

- 2-3 个子任务
- 结构清晰
- 风险较低
- 大多是简单串行或简单并行

例如：

- 先排序，再摘要
- 先检查，再补件
- 先归纳，再起草

这类任务应尽快支持，因为它们最接近真实工作，又最容易落地。

## L2：复杂复合任务

特点：

- 多条件分支
- 多业务对象
- 多系统依赖
- 风险更高

例如：

- “把这批客户里未来 30 天到期、风险变化明显、且最近未跟进的先筛出来，按优先级排序，再分别生成跟进建议，并对重大风险客户自动升级给风控”

这类任务更适合作为中后期能力，并需配合更严格治理。

---

## 七、什么时候不应该自动拆复合任务

并不是所有组合需求都适合自动拆解和自动执行。

以下几类场景应特别谨慎。

## 1. 涉及多个高风险动作

例如同时涉及：

- 风险定性
- 对外触达
- 流程提交
- 上报动作

这类任务即使要拆，也应以人工确认为前提。

## 2. 任务边界模糊

例如：

- “帮我把这个客户处理一下”

这种输入边界过于模糊，系统应先澄清，而不是直接拆。

## 3. 涉及跨岗位责任

例如一个输入里同时要求：

- 客户经理动作
- 审批岗动作
- 风控岗动作

这类需求未必适合由单个员工 Assistant 自动承接。

## 4. 输出目标不明确

如果系统无法判断：

- 最终交付物是什么
- 哪一步成功才算完成

那么也不适合自动拆解执行。

---

## 八、建议的五层任务系统升级版

如果产品要稳定支持复合任务，我建议把任务系统从原来的三层扩展成五层。

## 1. Task Template

标准任务模板。  
回答：

**这类岗位通常会做什么事。**

## 2. Task Detector

任务识别器。  
回答：

**当前是单任务还是复合任务。**

## 3. Task Planner

任务规划器。  
回答：

**如果是复合任务，应拆成哪些子任务，以及怎么组合。**

## 4. Task Instance Graph

任务实例图。  
回答：

**这一件具体事情现在由哪些子任务组成、各自状态是什么、依赖关系是什么。**

## 5. Task Aggregator

任务汇总器。  
回答：

**如何把多个子任务结果整理成统一交付。**

这五层一旦成立，系统就不再只是“识别某个任务模板”，而是真正具备了“理解复合工作需求”的能力。

---

## 九、最后的结论

如果用户输入涵盖多种任务类型的组合，系统不应：

- 强行只匹配一个任务模板
- 也不应直接把它视为一个全新的自由任务

更合理的做法是：

1. 把它识别为复合任务
2. 拆解成多个标准子任务
3. 生成任务图
4. 由 orchestration 引擎按依赖关系执行
5. 最终汇总成统一结果

如果把这篇文档压缩成最短表达，那么就是：

**用户提的是一个需求，系统内部跑的是一个任务组合。**

而这正是银行 AI Assistant 从“模板匹配工具”走向“任务编排工作代理”的关键一步。
