# Banking OpenClaw MVP Frontend Engineering Checklist

## 1. 目标

这份 checklist 面向 `apps/web`，用于约束 MVP 前端代码实现方式，确保工作台和后台管理页都保持轻量、清晰、可维护。

---

## 2. 必守规则

### 2.1 单文件行数限制

**每个代码文件不能超过 `500` 行。**

执行建议：

- 文件接近 `350-400` 行时主动拆分
- 页面拆成：
  - route/page
  - feature section
  - hooks
  - api client adapters
  - view-model helpers
- 不允许把整页 UI、状态逻辑、接口调用、表单逻辑全部堆在一个文件

### 2.2 前端不是业务真相源

必须遵守：

- 前端只消费后端权威对象
- 前端不维护正式任务状态机
- 前端不绕过后端直接访问企业系统

### 2.3 流式状态和落库事实分离

必须遵守：

- 流式输出是临时运行态
- API 查询结果是正式落库事实
- UI 上要能区分“生成中”和“已确认/已落库”

---

## 3. 页面与路由

### 3.1 路由按业务目的组织

推荐：

- `/workbench/session/:sessionId`
- `/tasks/:taskId`
- `/admin/roles`
- `/admin/task-templates`
- `/admin/labor`
- `/admin/substitution`

检查项：

- 每个页面只回答一个主要业务问题
- 管理页不要和员工工作页混成一个巨型页面
- 每个 route 文件只做页面装配，不承载过多逻辑

### 3.2 页面结构建议

推荐结构：

```text
page route
  -> page shell
  -> feature sections
  -> hooks
  -> query client calls
  -> pure render helpers
```

检查项：

- 页面容器负责拼装，不负责深业务计算
- 复杂表格、侧边栏、执行轨迹、分析卡片拆成独立组件

---

## 4. 状态管理

### 4.1 Server state 用 `TanStack Query`

适用对象：

- `Session`
- `Task`
- `Plan`
- `Execution`
- `LaborLedger`
- `SubstitutionScore`

检查项：

- 是否使用 query key 明确区分对象维度
- 是否让缓存失效策略显式可见
- 是否避免手写重复加载/重试逻辑

### 4.2 UI state 用 `Zustand`

适用对象：

- 当前侧边栏展开状态
- 当前 tab
- 本地筛选条件
- 临时对话框开关

检查项：

- store 不要塞入 server state 主对象
- 不要把 query 数据复制进 store 再维护一份

---

## 5. 组件设计

### 5.1 组件职责单一

检查项：

- 展示组件只负责渲染
- 容器组件负责组装数据
- 列表组件不要直接内嵌复杂业务规则

### 5.2 组件拆分信号

出现下面任一情况就应拆分：

- 文件超过 `350` 行
- 同时处理 3 类以上业务区块
- 同时管理 5 个以上局部交互状态
- 同时发起多个不相关数据请求

### 5.3 表格与表单分离

检查项：

- 复杂后台表格与编辑表单不要写在同一文件
- 表单 schema、默认值、提交逻辑可以抽到独立模块

---

## 6. API 交互

### 6.1 使用统一 client 层

检查项：

- 不在组件里散落 `fetch`
- API 调用统一走 client / query wrapper
- 参数与返回值使用共享 schema 或类型

### 6.2 错误处理统一

检查项：

- 查询错误有统一 fallback
- 表单提交错误有可见反馈
- 流式连接异常有重连/提示策略

---

## 7. 工作台特有要求

### 7.1 Session 页面

检查项：

- 消息流、计划结果、执行状态分区展示
- 用户输入、模型流式输出、正式任务对象不要混在一个列表语义里

### 7.2 Task 页面

检查项：

- 当前 `Plan`
- 当前 `Execution`
- 审批状态
- 工具调用摘要
- 审计摘要

应能快速看出：

- 现在执行到哪一步
- 是在等待审批还是等待工具结果
- 哪些信息是权威事实

### 7.3 Admin 页面

后台页面优先服务于：

- 任务测绘
- 劳动记账
- 替代分析

检查项：

- 页面支持校对而不是只展示结果
- 关键指标能追溯来源

---

## 8. UI 与交互

### 8.1 先保证清晰，再追求炫技

检查项：

- 状态标签统一
- 进度、审批、失败、回退语义明确
- 不用复杂动效掩盖状态不清

### 8.2 设计系统保持轻量

检查项：

- 复用 `shadcn/ui` 或轻量 design system
- 不为 MVP 建重型自研组件平台

---

## 9. 测试重点

优先覆盖：

- 会话主流程
- 任务详情主流程
- 后台关键列表和详情页
- 流式状态切换
- 审批等待与失败回退展示

避免：

- 大量低价值 snapshot 测试
- UI 文案微调引发大面积脆弱测试

---

## 10. 前端反模式

- 单页面文件超过 `500` 行
- 页面组件里直接写大量接口逻辑
- query 数据与 store 数据双份维护
- 把流式输出直接当最终结果
- 用前端逻辑偷偷决定业务状态
- 管理页与工作台职责混乱

---

## 11. 一句话结论

**前端 MVP 的最佳实践是：页面按业务目的组织，server state 与 UI state 分离，流式运行态与正式事实分离，并严格遵守单文件不超过 `500` 行。**
